Encapsulation

Encapsulation is the idea of wrapping data (attributes) and code acting on the data (methods) together as a single unit. In doing so, the internals of our class are hidden or protected from outside misuse or interference. Encapsulation is an effective strategy to reduce code complexity and prevent improper access across objects. To better understand this concept, let's delve into an example regarding my bank account: 


            public class BankAccount {

                public String bankName; // Why is this public?
                private String password; // But this private?
                private double balance;

            }

Although we have not addressed it yet, the presence of modifiers such as public and private are everpresent in our code and play a large role in upholding proper encapsulation. While there are multiple access modifiers (public, private, package-private, and protected), we will focus on public and private for purposes of this tutorial. Simply put, public allows any other class to access the particular attribute or method we are access modifying. Conversely, private restricts access to within the class, acting much more restrictive. In our example above, we set bankName to public but password and balance to private. Can you think why this might be the case? Let's take a look at a use case to better understand: 


            public static void main(String[] args) {

                BankAccount myCheckingAccount = new BankAccount();
                myCheckingAccount.bankName = "Wells Fargo"; // This attribute has public access so this is fine
                myCheckingAccount.password = "password123"; // ERROR: We can't access a private attribute!
                myCheckingAccount.balance = 1000000000.00; // ERROR: If we didn't protect access, you could create money out of thin air 

            }

Please note that the example we are using is not necessarily indicative of effective OOP or well thought out encapsulation but rather an example to demonstrate the base concept. Our bankName is not particularly significant in comparison to our password and balance attributes, which are highly sensitive in nature. If we had set those attributes to public, anyone that had access to our BankAccount instance could change the password and account balance to their liking. Proper encapsulation is an extension of abstraction, hiding implementation details within a class; as long as the class works as we would expect, the specifics behind its implementation are not important to us. Before we wrap up this topic, let's take a look at getters and setters, methods that allow us to access data without breaking encapsulation. 


            public double getBalance() {
                return balance; // Allows us to access our account balance without being able to change it directly
            }

If we wanted to see how much money we had in our checking account but our balance was set to private, we could use a method like getBalance(). This method, called a getter, allows us to obtain our current balance while still protecting our balance attribute from any interference or misuse. The opposite, a setter, allows us to set attributes without directly altering the instance's attributes. The use cases for these methods vary but they demonstrate proper data protection and encapsulation within our classes.
