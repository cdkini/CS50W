Abstraction

Abstraction is the idea of templating or defining the details of an object without delving into the specifics regarding implementation. For all intents and purposes, abstraction is creating a blueprint and allowing the user to fill in the specifics as he/she deems fit. Let's take a look at this idea by modeling Cars and Ferraris: 


            // Our template of what a car should do
            public interface Car {

                void drive();
                void brake();
                String getMake();
                String getModel();

            }      

            // Our specific implementation of a type of car
            public class Ferrari implements Car {
                
                private String model;
                private boolean inMotion;

                public Ferrari(String model) {
                    this.model = model;
                    this.inMotion = false;
                }

                public void drive() {
                    inMotion = true;
                }

                public void brake() {
                    inMotion = false;
                }

                public String getMake() {
                    return "Ferrari";
                }

                public String getModel() {
                    return model;
                }
            
            }

Immediately, we can see that this example uses a new piece of syntax: the concept of an interface. Interfaces are Java's answer to abstraction, allowing the user to give a high level overview as to what an object should be able to do. All cars, regardless of their type, price, or any other relevant variable, have the ability to drive and brake. Additionally, they all have makes and models that we should be able to extract. Note that our interface only incudes method signatures (the first line of a method); actual classes need to fill in the details.

Let's move on to our Ferrari class. Note that the first line, public class Ferrari implements Car contains yet another new keyword. Simply put, implements is used to state that our class will fill in the blanks left by our method signatures set by the relevant interface. Due to this keyword, we must fill in the details we passed on during our creation of the interface, writing proper drive(), brake(), getMake(), and getModel() methods. If we do not define all the methods noted in the interface, our Ferrari class will not compile. Can you think why this must be the case? Java allowed us to define what a Car is at a high level and because we've stated that all Cars must be able to do Car related things, our Ferrari class, which implements this interface, needs to do all Car related things.
